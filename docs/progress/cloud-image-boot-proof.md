# Hosted cloud image boots + control plane up (C2)

- **Specs touched:** `ENVIRONMENT.md` (# Provisioning, # Networking & discovery, # How the profile is realized), `BUILD.md` (# 2 disk images, # 6 qcow2), `TESTING.md` (# Full-stack control-plane integration) — all implemented, not changed.

C2 of the cloud-VM track in #196, after [hosted-cloud-image.md](hosted-cloud-image.md) (C1b — the lean image definition), [slim-cloud-host-agent.md](slim-cloud-host-agent.md) (C1c — the `-tags hosted` host-agent), and [hosted-setup-gate.md](hosted-setup-gate.md) (C3a — the brain-side `/setup` gate). It turns the lean image *definition* into a **bootable** image and proves the control plane cold-boots and serves under QEMU — no swtpm, no LUKS, no installer. The cloud analogue of the medium lane's full-stack boot, minus everything the hosted profile drops.

## What was done

- **Bake the control plane into the hosted image (`dev/cloud/bootstrap.sh`, extended).** C1b's bootstrap built the lean raw + asserted the cut list; C2 extends it to a full bootable bake, reusing the medium lane's mechanism: build the slim `-tags hosted` host-agent, `make control-plane-images` → the four `docker save` tarballs + `load-control-plane-images.sh` + `malmo-load-images.service` (first-boot `docker load`, offline-first, **reused as-is** from `dev/test-qemu/`), and stage the control-plane `compose.yml` + `caddy.json` at `/var/lib/malmo/control-plane/` (same-path constraint). Closes with the existing lean + `hosted`-marker assertions and a **raw → qcow2** `qemu-img convert` for the cloud artifact (`BUILD.md` # 6). `make build-cloud-image` now yields the full bootable image (its C1b lean-only form was a stepping stone); `make test-cloud-qemu` builds + boots it.
- **Generated ExtraTree kept separate (`dev/cloud/mkosi.extra.generated/`, gitignored).** mkosi overlays two `ExtraTrees`: the **committed** `mkosi.extra/` (just the static `/etc/malmo/profile` = `hosted` marker, untouched) and the generated tree bootstrap.sh populates (units, the slim host-agent binary + its brain-bootstrap drop-in, the image tarballs, the staged compose, the PAM service, the assertion harness). Keeps the committed marker clean in git.
- **Single NIC without NetworkManager (`dev/cloud/mkosi.postinst.chroot`, new).** Enables `systemd-networkd` + a DHCP `.network` matching the provider NIC (`en*`/`eth*`; Docker's own bridges never match) — the minimal cloud-native path (`ENVIRONMENT.md` # Networking & discovery), no NM/Avahi/sshd/TPM. Also enables docker + containerd + the image loader + host-agent + the assert oneshot, provisions the `malmo`/`malmo-app`/`malmo-shared` identities, and pre-commits machine-id (so the firstboot wizard can't block sysinit).
- **Minimal Go change so the *containerized* brain resolves `hosted` (`internal/hostagent/brainlaunch` + `cmd/host-agent-real`).** The brain reads its profile from `/etc/malmo/profile`, but its container mounts only the data root + socket dir, so without this it resolves `appliance` regardless of the image — and the hosted `/setup`→503 gate (C3a) would never fire in the booted image (C3a was only ever exercised by unit tests pointing `MALMO_PROFILE_PATH` at a temp file). `brainlaunch.Config` gains `ProfileMarker`; host-agent sets it to `/etc/malmo/profile` **only when that file exists on the host**, and `runSpec` then bind-mounts it read-only at the same path into the brain. An unmarked appliance box (and `make dev`) leaves it empty → no mount → brain defaults to `appliance`, byte-unchanged. This is the only Go change; it crosses C2's "no Go change" line by ~10 lines to satisfy C2's own security follow-up (see below) — the line assumed C1a/C1c had already wired the brain's hosted resolution end-to-end, which they hadn't for the containerized case.
- **In-VM assertions over the serial console, not SSH (`dev/cloud/cloud-assertions.sh` + `malmo-cloud-assert.service`, new).** The hosted image ships **no `openssh-server`** (lean cut list — C1b's bootstrap asserts its absence), so the medium lane's SSH-driven assertions don't apply. Instead a baked oneshot runs the assertions, writes a single `MALMO-CLOUD-ASSERT: PASS|FAIL` sentinel to `/dev/console`, and powers off; the host driver greps the captured serial log. The unit is **credential-gated** (`ConditionCredential=malmo.cloud-assert`) so it only fires when the harness injects that SMBIOS credential — a plain `make build-cloud-image` or a real cloud boot never runs it. Assertions (C2 scope): the `hosted` marker landed, systemd reached multi-user, **PSI is live** (`/proc/pressure/memory` non-empty — pairs with C1b's `psi=1`), Docker active, the four baked images loaded, the four control-plane containers running, the **socket-proxy boundary** held (brain has no raw `docker.sock`, `DOCKER_HOST=tcp://docker-proxy:2375`), the **dashboard SPA + `/api` reachable through Caddy**, and — the security pairing — **hosted unseeded `/setup` returns 503** (proving both that the open-`/setup` window stays shut in branch history *and* that the brain took the hosted seam via the marker mount).
- **Boot harness (`dev/cloud/run-cloud-tests.sh`, new) + `make test-cloud-qemu`.** Mirrors `run-medium-tests.sh` minus swtpm, the LUKS credential, and the two-boot cycle: one virtio NIC with `restrict=on` (air-gaps the guest, so a missing bundled image hard-fails — proving the offline bundle is complete; no SSH hostfwd needed), OVMF UEFI (reused probe), a single boot, serial capture + `dump_serial` on failure, and the SMBIOS credential that arms the assert unit. Waits for the sentinel, then for the VM's self-poweroff.

## What's next

- **VM-boot acceptance on the maintainer's env.** Per the medium-lane precedent (`TESTING.md`), VM-boot is the bar; it was **not run here** — `sudo make test-cloud-qemu` can't build/boot on this contributor box (the mkosi-26 / Ubuntu-24.04 `PR_CAPBSET_DROP` sandbox limitation, #189, same block the medium lane hits here). Everything else is verified: `make check`, `go vet`/`build -tags hosted ./...`, `bash -n` on every new script, and the brainlaunch unit tests for the marker mount.
- **C3a cloud-lane seed delivery (#220).** C2 boots with **no seed** (so `/setup`→503 is the unseeded gate). #220 delivers `seed.json` over an SMBIOS credential and asserts the seeded path end-to-end (correct secret → first admin, wrong/missing → 401, frozen identity across reboot). It builds directly on this slice's bake + the marker mount that makes the brain resolve `hosted`.
- **C4 first-run wizard (#208) and C5 cloud-image end-to-end (#209)** layer on top.
- **Known gaps:** the qcow2 carries the image id (`malmo-cloud-amd64.qcow2`), not a release version (`malmo-vX.Y.Z-amd64.qcow2`) — versioned naming is a release-time concern. `make build-cloud-image` now requires go + docker + qemu-img + libpam0g-dev (it bakes the control plane), heavier than C1b's mkosi-only lean build.
